# SSAFY RAG Chatbot Project

SSAFY 학습 도우미를 주제로 구현한 웹 기반 RAG 챗봇 프로젝트입니다.  
사용자는 웹 화면에서 질문을 입력할 수 있고, 서버는 OpenAI API를 호출하여 답변을 생성합니다. 또한 사용자가 업로드한 `txt`, `md`, `pdf` 문서를 기반으로 유사도 검색을 수행하여, 관련 문서가 있는 경우 해당 문서를 근거로 답변하는 RAG 기능도 포함하고 있습니다.

## 프로젝트 개요

이 프로젝트는 다음과 같은 흐름으로 동작합니다.

1. 사용자가 웹 화면에서 질문을 입력합니다.
2. 프론트엔드 JavaScript가 `/chat` API로 JSON 요청을 보냅니다.
3. 서버는 업로드된 문서들에 대해 유사도 검색을 수행합니다.
4. 관련 문서가 있으면 해당 문서를 근거로 OpenAI API를 호출합니다.
5. 관련 문서가 없으면 일반 생성형 AI 응답으로 답변합니다.
6. 응답 결과는 JSON 형태로 프론트엔드에 전달되고 화면에 출력됩니다.

## 주요 기능

### 기본 챗봇 기능

- HTML, CSS, JavaScript 기반 웹 채팅 화면
- `fetch` API를 이용한 `/chat` 비동기 요청 처리
- 사용자 질문과 AI 응답의 순차 출력
- 서버 오류 및 네트워크 오류 메시지 안내

### RAG 기능

- `txt`, `md`, `pdf` 문서 업로드 지원
- 업로드한 문서를 `servers/data` 디렉토리에 저장
- 문서 텍스트 추출 후 청킹
- 각 청크에 대한 임베딩 생성
- ChromaDB 기반 벡터 데이터베이스 저장
- 질문 임베딩과 문서 임베딩 간 유사도 검색
- 유사도 기준을 만족하는 문서만 답변 생성에 활용
- 문서 참고 여부 및 참고 문서명 응답 포함

## 실행 방법

### 1. 백엔드 환경 준비

```powershell
cd C:\ssafy15\chatbot-project_lab\servers
.\.venv\Scripts\python.exe -m pip install -r requirements.txt
```

### 2. 환경 변수 설정

`servers/.env.example`을 참고하여 `servers/.env` 파일을 작성합니다.

```env
OPENAI_API_KEY=your_openai_api_key_here
OPENAI_MODEL=gpt-4o-mini
RAG_MODEL_NAME=paraphrase-multilingual-MiniLM-L12-v2
```

### 3. 백엔드 서버 실행

```powershell
cd C:\ssafy15\chatbot-project_lab\servers
.\.venv\Scripts\python.exe .\main.py
```

서버가 실행되면 `http://localhost:8000`에서 API를 사용할 수 있습니다.

### 4. 프론트엔드 실행

`clients/index.html`을 브라우저에서 열거나 Live Server로 실행합니다.

## API 요약

### `POST /chat`

사용자 질문을 받아 RAG 검색 후 OpenAI 응답을 생성합니다.

요청 예시:

```json
{
  "message": "SSAFY 커리큘럼의 특징을 설명해 주세요."
}
```

응답 예시:

```json
{
  "answer": "설명 내용...",
  "sources": ["SSAFY_GUIDE.txt"],
  "used_rag": true,
  "reference_note": "업로드된 문서를 참고해 답변했습니다.",
  "similarity_threshold": 0.45,
  "chunk_size": 800,
  "chunk_overlap": 120,
  "provider": "openai"
}
```

### `POST /upload`

문서를 업로드하고 RAG 인덱스를 다시 생성합니다.

지원 파일 형식:

- `.txt`
- `.md`
- `.pdf`

### `GET /documents`

현재 RAG 대상 문서 목록과 청크 수를 조회합니다.

### `GET /health`

서버 상태, 현재 문서 수, 청크 수를 확인합니다.

먼저 사용자가 업로드한 문서는 `servers/data` 디렉토리에 저장됩니다. 서버는 이 문서들에서 텍스트를 추출하고, 일정 길이의 청크로 분할합니다. 각 청크는 임베딩 벡터로 변환된 뒤 ChromaDB에 저장됩니다. 사용자가 질문하면 질문도 동일한 임베딩 모델로 벡터화하고, 벡터 데이터베이스에서 유사한 문서 청크를 검색합니다. 이때 유사도 기준을 만족하는 청크만 선택되며, 선택된 청크가 있을 경우 그 내용을 OpenAI API 호출 시 시스템 프롬프트에 포함하여 답변을 생성합니다. 검색 결과가 없으면 일반 생성형 AI 응답으로 처리하며, 응답에는 문서 참고 여부가 함께 포함됩니다.

## 구현 결과 및 고찰

### 생성형 AI API 호출 흐름에 대한 이해

이번 프로젝트를 통해 생성형 AI API가 실제 서비스에서 어떤 흐름으로 호출되고 응답되는지 구체적으로 이해할 수 있었습니다. 사용자가 웹 화면에서 질문을 입력하면, 프론트엔드의 JavaScript가 `fetch` API를 통해 서버의 `/chat` API로 JSON 형식의 요청을 전송합니다. 서버는 이 요청에서 사용자의 질문을 받아 필요한 전처리를 수행한 뒤, 외부 생성형 AI API인 OpenAI API를 호출하여 답변을 생성합니다. 생성된 결과는 다시 JSON 형태로 프론트엔드에 전달되며, 화면에서는 이를 사용자 메시지 아래에 순차적으로 출력합니다. 이 과정을 구현하면서 생성형 AI 서비스는 단순히 모델을 호출하는 것이 아니라, 사용자 입력 처리, 서버와의 통신, 외부 API 호출, 응답 후처리, 화면 출력까지 연결된 전체 흐름으로 이해해야 한다는 점을 확인할 수 있었습니다.

### 문서 업로드 전후 AI 응답의 차이

문서 업로드 기능을 적용하기 전에는 AI가 사전학습된 일반 지식만을 바탕으로 답변을 생성하였습니다. 따라서 질문에 대한 전반적인 설명은 가능했지만, 프로젝트에서 다루는 특정 문서나 사용자가 직접 제공한 자료를 근거로 한 답변은 제공하지 못했습니다. 반면 문서를 업로드한 이후에는 해당 문서가 서버의 `data` 디렉토리에 저장되고, 문서 내용이 청킹 및 임베딩 과정을 거쳐 벡터 데이터베이스에 반영됩니다. 이후 사용자가 질문하면, 질문과 관련성이 높은 문서 청크를 먼저 검색한 뒤 그 내용을 AI 입력에 포함하여 답변을 생성하게 됩니다. 그 결과 업로드 전보다 훨씬 구체적이고 문서 중심적인 응답이 가능해졌으며, 동일한 질문이라도 업로드한 문서에 포함된 표현이나 핵심 내용이 답변에 반영되는 차이를 확인할 수 있었습니다.

### RAG 적용 시 장점과 한계

RAG를 적용한 가장 큰 장점은 모델이 사전에 학습하지 않은 정보나 최신 자료, 또는 특정 조직 내부 문서 내용을 답변에 반영할 수 있다는 점입니다. 즉, 모델 자체를 다시 학습시키지 않아도 사용자가 업로드한 문서만으로 새로운 지식을 즉시 반영할 수 있어 활용성이 높습니다. 또한 답변이 어떤 문서를 참고했는지 함께 제시할 수 있기 때문에, 단순 생성형 AI보다 답변의 근거를 설명하기 쉽고 신뢰도를 높일 수 있다는 장점도 있습니다. 그러나 한계 역시 분명히 존재합니다. 질문과 관련된 적절한 문서 청크를 검색하지 못하면 RAG의 효과는 거의 사라지며, 반대로 관련성이 낮은 문서가 검색되면 오히려 답변 품질이 저하될 수 있습니다. 또한 청킹 크기, 오버랩 크기, 유사도 기준값과 같은 설정에 따라 검색 결과가 달라지므로, 단순히 RAG를 적용했다고 해서 항상 좋은 결과가 나오는 것은 아니라는 점도 확인할 수 있었습니다.

### 데이터 품질이 응답에 미치는 영향

이번 구현을 통해 데이터 품질이 RAG 응답 품질에 매우 큰 영향을 준다는 점도 확인할 수 있었습니다. 텍스트가 명확하게 정리되어 있고 핵심 내용이 잘 드러나는 문서는 청킹 이후에도 의미 단위가 비교적 잘 유지되기 때문에, 임베딩 품질과 검색 정확도가 높아집니다. 반면 PDF에서 텍스트 추출이 깨지거나, 문서에 불필요한 정보가 많거나, 구조가 지나치게 복잡한 경우에는 검색 결과의 품질이 낮아질 수 있습니다. 이 경우 AI는 관련성이 낮은 문서 조각을 참고하거나, 문서 기반 답변을 제대로 생성하지 못할 수 있습니다. 따라서 RAG 시스템에서는 모델 성능만큼이나 문서의 정제 상태, 구조, 텍스트 추출 품질이 중요하며, 좋은 답변을 얻기 위해서는 입력 문서의 품질 관리가 반드시 필요하다고 생각합니다.

### 향후 개선 또는 확장 가능성

향후에는 여러 방향으로 개선 및 확장이 가능하다고 생각합니다. 먼저 현재는 문서를 업로드할 때마다 전체 인덱스를 다시 구성하는 방식이므로, 문서 수가 많아질 경우 비효율적일 수 있습니다. 이를 해결하기 위해 새 문서만 추가 반영하는 증분 인덱싱 방식을 도입할 수 있습니다. 또한 벡터 검색만 사용하는 현재 구조에서 더 나아가, 키워드 검색과 벡터 검색을 결합한 하이브리드 검색 방식을 적용하면 검색 정확도를 높일 수 있을 것입니다. 프론트엔드 측면에서는 어떤 문서의 어떤 부분이 실제로 참고되었는지 더 자세히 보여주는 기능을 추가하면 사용자 신뢰도를 높일 수 있습니다. 나아가 대화 이력을 고려한 멀티턴 챗봇 구조, 문서 요약 기능, 외부 에이전트 스킬과의 연동 기능 등을 추가한다면 단순한 문서 기반 QA 시스템을 넘어 보다 실용적인 AI 어시스턴트로 발전시킬 수 있을 것입니다.

## 참고 사항

- 실제 API 키는 `servers/.env`에만 저장하고 Git에는 포함하지 않습니다.
- 업로드한 문서와 ChromaDB 데이터는 `.gitignore`에 의해 제외됩니다.
- OpenAI API 키가 잘못되었거나 권한이 없는 경우 `/chat` 요청 시 `500` 오류가 발생할 수 있습니다.
